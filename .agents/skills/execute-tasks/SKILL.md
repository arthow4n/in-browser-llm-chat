---
name: execute-tasks
description: Shouldn't be invoked unless explicitly requested by the user. Sequentially executes tasks from a markdown file by spawning subagents for each step until completion or repeated errors occur.
---

You have been invoked to execute tasks from a tasks markdown file specified by the user (typically generated by the `breakdown-plan` and `review-implementation-tasks` skills).

Follow this process strictly:

1. Initialization:
   - Identify the tasks markdown file specified by the user.
   - Set the maximum consecutive error threshold X (default is 3, unless the user specifies otherwise).
   - Initialize a `consecutive_errors` counter to 0.

2. Execution Loop:
   - Read the tasks file to identify the next pending step. If all steps are clearly marked as done, break the loop and proceed to Completion.
   - Spawn an isolated subagent and instruct it to execute exactly 1 pending step from the tasks file. Instruct the subagent to:
     - Read the specific task description.
     - Implement the required changes.
     - Mark the step as completed in the tasks markdown file.
     - Git commit the changes with an appropriate commit message.
     - Alternatively, if the subagent determines that all steps are actually completed, it should explicitly report that back instead of doing any work.
   - Wait for the subagent to finish its task and return a response.
   - Evaluate the subagent's result:
     - If the subagent succeeded and marked a task as done, reset `consecutive_errors` to 0.
     - If the subagent reports that all steps are already completed, break the execution loop.
     - If the subagent encountered an error, failed to complete the task, or failed to follow instructions, increment the `consecutive_errors` counter.

3. Error Handling:
   - If `consecutive_errors` reaches X (default 3), halt the execution loop immediately and report the repeated failures to the user. Do not continue spawning subagents.

4. Completion:
   - Once the loop breaks (either due to all steps being completed or reaching the maximum error threshold), provide a summary of the progress and any remaining work or errors to the user.
   - If the loop exited because all steps were successfully completed, follow this 2-step process carefully to maintain git history:
     1. Move the tasks file to `docs.outdated` and git commit the file movement.
     2. Delete the newly moved tasks file in `docs.outdated` and make a separate git commit for the deletion. It is crucial to separate the movement and the deletion into 2 separate commits so that we can track the history of the outdated docs.
